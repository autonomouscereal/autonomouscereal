Agentic IT: what I am actually building

June 2026 / Cyril Blyseth

Digital work is convoluted to the nth degree. The answer is not another dashboard, chatbot, or one-off automation script.

This project is easy to misunderstand if you only look at a screenshot. It is not an iTop clone. It is not a Wazuh wrapper. It is not a SOC dashboard with a model bolted on. The dashboard is where humans can supervise the work, but the thing I am building is an autonomous enterprise operations layer.

The goal is local/private by default: install the operating layer, connect it to the tools an organization already has, deploy approved reference modules where there are gaps, and let agents take on real operational work with approvals, scoped access, evidence, user follow-up, and postmortems.

Why I care about this problem

I have been inside the kind of environment this is meant to reduce. Network monitoring, security incidents, phishing investigations, Splunk and Palo Alto logs, firewall changes, DNS records, case notes, change boards, CI/CD security checks, XSOAR-style automation, user support, team handoffs, and the constant problem of getting the right context to the right person at the right time.

The work is not hard because one ticket is impossible. It is hard because every ticket crosses tools, teams, permissions, calendars, policies, logs, and half-remembered tribal knowledge. A model answering in chat does not fix that. A hardcoded playbook fixes one slice of it. The interesting problem is turning messy human work into governed agent work without losing the operational record.

What I am trying to build

The system should be able to start small, but it should still be built like real infrastructure. The rough setup:

  • Install one private operating layer.
  • Discover or connect to the enterprise tools already in place.
  • Deploy reference modules only where the organization wants the platform to fill gaps.
  • Turn tickets, alerts, chat, email, CI/CD events, schedules, and direct requests into structured work.
  • Assign that work to agents with narrow permissions and clear boundaries.
  • Stop for approval before risky or environment-changing action.
  • Write evidence as the work happens, not as a cleanup step after everyone forgot the details.
  • After completion, run postmortems that can create or improve workflows, skills, tests, and knowledge.

The end state is not "AI helps with tickets." The end state is that a small number of humans supervise approvals, exceptions, strategy, and accountability while agents handle more and more of the repeatable operational labor.

Why the first proof is SOC and IT

SOC and IT are not the final boundary. They are the first useful proof because they exercise nearly every hard primitive: alerts, tickets, identity, email, logs, endpoint signals, code scanning, change requests, user questions, false positives, evidence, escalation, and remediation.

The current reference environment has included iTop, Wazuh, Zeek, Suricata, Mailcow, report-phish flows, Keycloak, GitLab, Semgrep, Trivy, OWASP ZAP, Nuclei, SearXNG, local/cloud model routing, and agent harness work. Those tools are not the product. They are the lab proving the operating loop.

The operating loop

A user should not need to know whether their issue is an incident, service request, access request, IAM problem, CI/CD problem, or security alert. They should describe the problem. The intake agent should ask concise clarifying questions when needed, classify urgency, look for related tickets or outages, route through assignment logic, and create or update the right work record.

Then the ticket agent works the issue. It reads the context, checks whether there is an approved workflow or knowledge article, decides what tools and evidence it needs, requests access if it hits a permission wall, requests approval if it needs to change something, updates the user when it is waiting, and completes or stages the next action.

After that, a postmortem pass asks the part most automation skips: should this become a reusable workflow, should an existing workflow change, did the alert need tuning, did the knowledge base miss something, did the agent need a new skill, and what test would catch this next time?

Guardrails are the architecture

The agent should be allowed to reason. The platform should enforce the walls.

A ticket agent can request access, but it cannot grant access to itself. It can request a change, but it cannot approve risky work. It can gather evidence, but it should not hide product-specific failure behind a friendly status message. It can continue after a user responds, but the continuation needs to know what already happened.

That means approval gates, scoped credential leases, access requests, provider sync, tool health, agent logs, ticket notes, and audit events are not side features. They are the difference between a neat demo and something I would trust near real infrastructure.

Why this belongs in AI infrastructure

This work is not just "add AI to ITSM." It touches the parts of AI systems that I think matter most going forward: local model routing, harness behavior, memory scoping, tool permissioning, workflow evaluation, agent continuation, run replay, provider adapters, and learning from completed work without turning every environment into a SaaS data source.

The question I keep chasing is practical: how much real enterprise work can agents take over if the system around them gives them the right context, narrow tools, approval paths, memory, and a way to learn from the work they finish?
