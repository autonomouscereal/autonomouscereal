Void: local AI execution notes

June 2026 / Cyril Blyseth

Void is older/superseded work now. I am keeping this note because it shows how I was thinking about local AI execution: tools, traces, job records, artifacts, and review loops.

The system is what happens after the model says something: tools, files, jobs, manifests, quality checks, failures, retries, and evidence.

I built Void because I kept running into the same problem: local AI stacks can generate impressive outputs, but the work around the model turns into a pile of brittle scripts. One script calls an image workflow. Another calls a video workflow. Another searches the web. Another writes a file. Nothing shares a job model. Nothing shares evidence. Nothing is easy to rerun.

That is fine for a one-off demo. It is not fine if the goal is an execution layer for local and self-hosted AI.

The execution layer

I still like an OpenAI-compatible endpoint because it gives tools and apps a familiar front door. But the front door is not the architecture. The important part is what the orchestrator does after the request arrives.

In Void, the core idea was to route work through strict envelopes and job records:

  • Normalize assistant and tool output before trusting it.
  • Separate chat responses from tool execution.
  • Give long-running jobs stable IDs and lifecycle state.
  • Stamp media and code work with seeds, parameters, hashes, and manifests.
  • Preserve traces that can be reviewed, replayed, or used for later training.

The model can still be creative. The system around it has to be strict about what actually happened.

Why envelopes matter

A model can say "I used the image tool" in plain text. That is not a tool call. It is a claim. If the orchestration layer treats claims as actions, the whole stack is not reliable enough for real work.

I want a stricter boundary: 

request
  -> planner / semantic router
  -> typed tool envelope
  -> tool execution
  -> artifact manifest
  -> quality review
  -> trace record
  -> user-facing result

The envelope is the contract between model language and system behavior. It is where arguments get normalized, malformed JSON gets rejected or repaired, tool names get resolved, and the work becomes something the rest of the system can reason about.

This is especially important for multimodal work. Image, video, music, TTS, OCR, VLM, RAG, research, and code loops all have different runtimes and failure modes. If they all return random strings, the system cannot make good decisions afterward.

Local-first does not mean small ambition

Local AI is not just about avoiding API bills. The real reasons are privacy, latency, hardware control, offline work, reproducibility, and the ability to build the pipelines you need without waiting for a hosted product to expose the exact capability.

But local-first makes orchestration harder. You have to think about:

  • Which services are running.
  • Which models are loaded.
  • What hardware is available.
  • How long jobs can run.
  • Where artifacts are stored.
  • How quality is judged.
  • How to recover after a service fails halfway through.

That is why Void treated media generation, research collection, code loops, RAG hygiene, artifact indexing, and quality review as part of one execution layer. The point was not to have every tool. The point was to make tools composable without losing evidence.

Quality review is part of the runtime

A lot of AI systems stop at generation. I think that is the wrong stopping point. If a system creates an artifact, it should also create enough context to decide whether the artifact is any good.

For media, that can mean scores, review committees, locks, post-run review, style packs, seed tracking, and refinement decisions. For research, it can mean sources, extraction notes, contradictions, and confidence. For code, it can mean tests, diffs, lint results, and human approval.

The pattern is the same: generation is not done until the system can say what happened and why the result should or should not be trusted.

The real product is evidence

The deeper I get into local AI systems, the more I think the winners will not just have better prompts or prettier frontends. They will have better execution records.

The serious question is:

  • Can it do the work again?
  • Can it explain what tools ran?
  • Can it show what files changed?
  • Can it preserve the seed and parameters?
  • Can it fail without corrupting the workspace?
  • Can the trace become training data later?

That is the layer I kept building toward after Void: local AI that does real work and leaves enough evidence to improve itself.
