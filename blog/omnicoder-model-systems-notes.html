OmniCoder: model systems notes

June 2026 / Cyril Blyseth

OmniCoder asks one main question: how much can a compact model family do when routing, memory, verification, and export are designed together?

OmniCoder is not an app wrapper and it is not a claim that I have a finished frontier model sitting in a public repo. It is an architecture lab for an omnimodal model family: one codebase exploring how text, code, images, video, audio, structured context, and tool/action traces can move through a shared reasoning system.

The practical bet is that smaller local-capable models can become much more useful when routing, memory, verifier loops, multimodal tokenization, and runtime export are treated as first-class design constraints instead of cleanup work after the model exists.

The shape of the system

The architecture breaks down into four layers:

  • Modality adapters and tokenizers for text/code tokens, image/video/audio VQ codes, continuous latents, metadata, and tool traces.
  • A shared reasoning core built around sparse experts, routing, long-context memory, retrieval hooks, and refinement.
  • Output heads and decoders for text, code, image, video, audio, verifier outputs, action/tool heads, and latent refinement.
  • Verification, training, and runtime packaging so the system can be tested, distilled, exported, benchmarked, and deployed.

The important part is not that every modality magically becomes the same thing. The important part is that each modality has a way into and out of the same reasoning core.

Routing is the model doing less on purpose

A model that can handle many modalities cannot afford to wake every capability for every request. OmniCoder uses sparse MoE, hierarchical routing, expert paging hooks, variable-K routing, capacity-aware dispatch, and runtime switches because the routing decision affects latency, memory, batching, export, and specialization.

If the request is mostly code, the system should not pay the full cost of image, video, and audio paths. If the request is visual, it should route toward the right representation path. If the target runtime is a phone, a laptop, or a constrained local box, the model has to respect the actual budget.

Long context is a runtime problem

OmniCoder treats context length as a systems problem, not as a number to brag about. The repo includes sliding-window decode, memory slots, landmark/random-access attention work, retrieval and PQ/kNN hooks, KV quantization, KV retention sidecars, and learned KV compression work.

The point is bounded usefulness. What should stay hot? What should be compressed? What should be retrieved? What can be dropped? That decision belongs to the model and runtime together.

Multimodal output is the hard part

Feeding an image into a model is only one side of the problem. The harder question is how the system should produce useful outputs across text, code, images, video, audio, and tool actions without turning into a pile of unrelated services.

That is why OmniCoder has scaffolding for image VQ, image decoding, video VQ, video heads, interpolation, audio tokenization, audio VQ-VAE, vocoder paths, ASR/TTS adapters, grounding, cross-modal fusion, verifier heads, and tool/action heads. Some pieces are runnable smokes. Some pieces define interfaces for future training and weights. The value is in forcing the interfaces to exist.

Verification is part of the model

I do not want a system that just emits a confident answer. OmniCoder includes HRM-style refinement, reward modeling, GRPO/PPO/RLHF scaffolds, code verification, multi-solution generation, verifier distillation, cross-modal verification, and cycle-consistency ideas because candidate generation and checking should be designed together.

Code should be tested. Tool actions should be validated. Cross-modal conversions should have ways to catch obvious failures. Visual and audio outputs should be checked against constraints instead of treated as successful because the output file exists.

Export keeps the research honest

Runtime export is not a side quest in this project. ONNX, Core ML, DirectML, NNAPI-style runners, ExecuTorch, llama.cpp/GGUF, MLC/TVM hooks, provider benches, mobile presets, and package manifests matter because deployment constraints should shape the model architecture.

If the model can only work in one giant training environment, it does not answer the local/private/edge question I care about. Export paths reveal where the design is real and where it is just a nice idea in a Python file.

What the repo is right now

The public repo is a research base. It does not publish the full model weights or every private training run. It does contain the architecture lanes, tests, small canaries, export tools, provider benches, runtime adapters, training plumbing, and enough scaffolding to make the research direction inspectable.

That is why I call it model systems work. The model, router, memory, verifier, data path, tool path, and runtime cannot be designed separately if the goal is a useful omnimodal system that can run outside a lab.
